# ReadOnlyTag: правила использования

Документ является рабочим brief для обновления Storybook-документации `ReadOnlyTag`.
Он фиксирует назначение компонента, границы поведения и правила контента на основе текущей реализации UFO и внешних дизайн-систем.

## Источники

- [eBay Playbook: Signal](https://playbook.ebay.com/design-system/components/signal)
- [LINE Design System: Tag](https://designsystem.line.me/LDSG/components/contents/tag-en)
- [Ahoy: Tag](https://ahoy.teamleader.design/3547946e7/p/19779f-tag)
- [Ant Design: Tag](https://ant.design/components/tag)
- [Atlassian Design System: Tag](https://atlassian.design/components/tag/examples)
- [Carbon Design System: Tag](https://carbondesignsystem.com/components/tag/usage/)

## Назначение

`ReadOnlyTag` - статичная информационная метка для коротких атрибутов, категорий и вспомогательных признаков, которые нужно только прочитать. Компонент помогает быстро считать дополнительный контекст рядом с карточкой, строкой таблицы, объектом списка, изображением или другим UI-элементом.

`ReadOnlyTag` не является действием, ссылкой, фильтром, checkbox, статусом процесса или удаляемым значением. Если тег должен менять состояние интерфейса, быть выбранным, вести по ссылке или удаляться пользователем, нужен другой компонент семейства Tag.

## Выводы из референсов

- Carbon, Ant Design и Atlassian разделяют теги по поведению: обычные информационные метки, selectable/filter tags, closeable/dismissible tags и action/link-варианты не должны смешиваться в одном read-only компоненте.
- LINE описывает базовую анатомию тега как optional icon, container и text. Ширина должна подстраиваться под длину текста, а высота и внутренние отступы должны оставаться стабильными.
- eBay Signal полезен как ограничение: декоративная метка не должна заменять полноценный статус, предупреждение или критически важный сигнал.
- Ahoy размещен на Zeroheight. Публичный HTML показывает, что страница требует доступа к содержимому, поэтому Ahoy не используется как источник конкретных правил без предоставленного текста или доступа.

## Базовые правила

- Используйте `ReadOnlyTag` только для информации, которую пользователь читает, но с которой не взаимодействует.
- Держите текст коротким: обычно 1-3 слова или компактное значение.
- Используйте цвет для визуального различения групп, но не как единственный источник смысла.
- Не добавляйте поведение через внешние обработчики клика, `tabIndex`, роли кнопки или checkbox.
- Не используйте `ReadOnlyTag` как единственный способ сообщить критически важную информацию.
- Не перегружайте интерфейс множеством разноцветных тегов рядом: теги должны помогать сканировать контекст, а не спорить с основным содержанием.

## Когда использовать

- Короткая категория: `Новинка`, `Популярное`, `Серия A`.
- Компактный атрибут объекта: `До 5 кг`, `4 товара`, `Онлайн`.
- Дополнительное свойство карточки, товара, строки таблицы или списка.
- Нейтральная или уточняющая метка, которая не требует действия.
- Визуальная группировка информации, если текст метки остается понятным без цвета.

## Когда не использовать

Не используйте `ReadOnlyTag`, если элемент должен:

- работать как кнопка, ссылка или короткое действие;
- открывать страницу, модальное окно, popover или меню;
- быть выбранным или снятым с выбора;
- работать как фильтр, checkbox или radio;
- удаляться через крестик или кнопку закрытия;
- показывать полноценный статус процесса, ошибку, предупреждение или успех;
- заменять alert, notification, badge, status label или системное сообщение;
- содержать длинный текст, который важен без обрезки.

## Связанные компоненты

| Компонент | Когда использовать |
| --- | --- |
| `ReadOnlyTag` | Статичная короткая информация без взаимодействия. |
| `SelectableTag` | Выбор значения с поведением checkbox. |
| `DismissibleTag` | Значение, которое пользователь может удалить. |
| `OperationalTag` | Компактное действие или ссылка. |

Не заменяйте один тип тега другим, если меняется поведение. Внешний вид может быть похожим, но семантика и доступность должны соответствовать задаче.

## Анатомия

Текущая реализация `ReadOnlyTag`:

```text
ReadOnlyTag
└── root span
    └── container
        ├── leading icon, optional
        └── text wrapper
            └── label
```

- Root - статичный `span`, который не попадает в tab order.
- Container - фон, скругление, высота, горизонтальные отступы и обрезка содержимого.
- Leading icon - необязательная декоративная иконка перед текстом.
- Text wrapper - внутренние отступы текста.
- Label - короткое значение тега из `label` или default slot.

## Публичный API

| API | Тип | По умолчанию | Описание |
| --- | --- | --- | --- |
| `label` | `string` | `'Tag'` | Текст тега. Может быть заменен содержимым default slot. |
| `size` | `'tiny' \| 'small' \| 'medium' \| 'large'` | `'tiny'` | Размер тега: высота, типографика, отступы и размер leading icon. |
| `color` | `'gray' \| 'blue' \| 'purple' \| 'green' \| 'red' \| 'orange' \| 'orange-accent' \| 'yellow'` | `'gray'` | Цветовая схема для визуального различения информационных меток. |
| `default` slot | slot | - | Текст или простое содержимое вместо `label`. |
| `leading` slot | slot | - | Необязательная декоративная иконка перед текстом. |

## Не поддерживается

`ReadOnlyTag` не поддерживает и не должен имитировать:

- `href`;
- `onClick` и другие action callbacks;
- `selected`;
- `checked`;
- `disabled`;
- `onClose`;
- `onDismiss`;
- keyboard handlers;
- `role="button"`;
- `role="checkbox"`;
- `aria-pressed`;
- `aria-checked`;
- ручной `tabIndex` для фокусировки.

Если одно из этих свойств нужно продуктово, выберите другой компонент семейства Tag или отдельный компонент под нужную семантику.

## Варианты цвета

Цветовые варианты нужны для мягкого визуального различения информационных меток:

| Color | Назначение |
| --- | --- |
| `gray` | Базовая нейтральная метка без смыслового акцента. |
| `blue` | Информационная или уточняющая метка. |
| `purple` | Дополнительный акцент для вторичной классификации. |
| `green` | Положительный или подтверждающий информационный оттенок, но не полноценный статус успеха. |
| `red` | Критический или ограничивающий информационный оттенок, но не alert и не ошибка формы. |
| `orange` | Светлый брендовый акцент. |
| `orange-accent` | Контрастный брендовый акцент с инверсным текстом. |
| `yellow` | Предупреждающий информационный оттенок, но не системное предупреждение. |

Правило: если смысл исчезает без цвета, добавьте смысл в текст или используйте более подходящий компонент.

## Размеры

| Size | Высота | Иконка | Текст | Использование |
| --- | --- | --- | --- | --- |
| `tiny` | 24px | 16px | 14px / 130% | Самые плотные интерфейсные зоны и компактные списки. |
| `small` | 28px | 18px | 14px / 130% | Компактная дополнительная информация. |
| `medium` | 32px | 18px | 16px / 140% | Базовый размер для большинства случаев. |
| `large` | 36px | 20px | 16px / 140% | Более заметные метки в свободных layout-зонах. |

Ширина зависит от содержимого. Максимальная ширина текущей реализации - 256px. Длинный текст остается в одну строку и обрезается через ellipsis.

## Поведение и состояния

`ReadOnlyTag` имеет только статичное состояние отображения.

- Нет hover-состояния.
- Нет active или pressed-состояния.
- Нет focus-visible-состояния.
- Нет disabled-состояния.
- Нет selected или checked-состояния.
- Нет loading, invalid или error-состояния.
- Нет pointer и keyboard interaction.

Если в макете нужно показать disabled, selected, checked или dismissible behavior, это признак другого компонента, а не состояния `ReadOnlyTag`.

## Правила контента

- Пишите коротко и конкретно: `Новинка`, `Онлайн`, `До 5 кг`, `4 товара`.
- Не ставьте точку в конце.
- Не используйте текст действия: `Перейти`, `Подробнее`, `Нажмите`.
- Не вкладывайте сложную разметку, интерактивные элементы, ссылки или кнопки в default slot.
- Используйте leading icon только если она ускоряет распознавание или поддерживает текстовый смысл.
- Не полагайтесь на иконку как на единственный носитель смысла: смысл должен быть понятен из текста.

## Доступность

- Семантический элемент: `span`.
- Роль: не задается.
- Клавиатура: компонент не фокусируется и не управляется с клавиатуры.
- Фокус: собственного focus-visible состояния нет.
- ARIA-состояния: `aria-pressed`, `aria-checked`, `aria-selected` и похожие состояния не применяются.
- Disabled: не применяется. Компонент либо отображается, либо не отображается.
- Screen reader: читает текстовое содержимое тега как обычный текст в потоке.
- Leading icon: декоративная обертка скрыта через `aria-hidden="true"`.

Критически важная информация должна быть продублирована в основном тексте интерфейса или выражена специализированным компонентом статуса, предупреждения или ошибки.

## Карта для Storybook

При переносе brief в Storybook используйте эту структуру:

- `Playground` - интерактивная проверка `label`, `size`, `color`.
- `Sizes` - все размеры рядом, чтобы проверить высоту, типографику и иконку.
- `Colors` - все цветовые варианты с русскими примерами.
- `WithIcon` - пример с `leading` slot.
- `LongContent` - обрезка длинного текста через ellipsis.
- `Usage` - хорошие и плохие примеры применения.
- `Accessibility` - статичная семантика, отсутствие tab order и ARIA-состояний.
- `ThemeBlock` - проверка светлой и темной темы, если история уже используется.

Не добавляйте stories для disabled, selected, checked, click или keyboard behavior, потому что такого поведения нет в реализации.

## Критерии готовности Storybook-документации

- Документация написана на русском языке и сохранена в UTF-8 без BOM.
- Текст не содержит mojibake и символов замены вместо кириллицы.
- Props, slots и states соответствуют фактическому API `ReadOnlyTag`.
- Документация ясно отделяет `ReadOnlyTag` от `SelectableTag`, `DismissibleTag` и `OperationalTag`.
- Цвет не описан как единственный источник смысла.
- Доступность описана как статичный текстовый элемент без интерактивной семантики.

## Проверка после переноса в Storybook

Для самого Markdown brief отдельные unit-тесты не нужны.

После обновления Storybook-документации запустите:

```bash
npm run typecheck
npm run build-storybook
```

Если вместе с документацией меняется компонент или stories с логикой, дополнительно запустите:

```bash
npm run test
```
